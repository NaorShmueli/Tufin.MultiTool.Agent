# Tufin.MultiTool.Agent

Multi-tool AI agent REST API with structured observability. The service accepts a natural-language task, lets a local LLM decide which tools to call, executes those tools in the backend, stores every observable step in SQLite, and returns the final answer plus the full trace.

## Features

- `POST /task` submits and executes a task.
- `GET /tasks/{taskId}` retrieves a previous task with its trace.
- `GET /tasks` lists recent tasks.
- `GET /health` returns service health.
- Persistent SQLite observability store.
- Structured trace events for task start, model decisions, tool calls, tool results, final answer, and completion/failure.
- Per-task latency and token usage.
- Local LLM support through Ollama.
- Real backend tools: `calculator`, `weather`, `unit_converter`, `database_query`.

## Reasoning Loop

The agent uses an observe-decide-act loop:

1. The API creates a task record with status `Pending`.
2. `AgentRunner` starts the task and records `TaskStarted`.
3. The current conversation and tool definitions are sent to Ollama.
4. The model either returns tool calls or a final answer.
5. Every model response is recorded as a `ModelDecision` event with latency and token usage.
6. Tool calls are validated against the registered backend tools.
7. Each tool request is recorded as `ToolCall`.
8. The backend executes the real tool function.
9. Each result is recorded as `ToolResult` with latency and output/error.
10. Tool results are sent back to the model for interpretation.
11. The model produces a final user-facing answer.
12. The final answer and completion state are persisted.

The final answer is intentionally generated by the model after observing tool results. Raw tool output is not returned directly as the final response.

## Tools

| Tool | Description |
|---|---|
| `calculator` | Safely evaluates arithmetic expressions with operators such as `+`, `-`, `*`, `/`, `%`, `^`, and parentheses. |
| `weather` | Fetches current weather from Open-Meteo for a requested city. |
| `unit_converter` | Converts length, weight, and temperature units. |
| `database_query` | Executes safe read-only `SELECT` queries against a pre-seeded fictional product catalog SQLite database. |

## Catalog Database Bonus

The `database_query` tool is backed by a separate SQLite database, not the observability database.

```text
data/catalog.db
```

Seeded schema:

```sql
products(id, name, category, price, stock_quantity)
orders(id, customer_name, order_date, status)
order_items(id, order_id, product_id, quantity, unit_price)
```

The database includes fictional products such as `iPhone 17`, `iPhone 17 Pro`, `MacBook Air 15`, accessories, monitors, and sample orders.

Safety rules:

- Only single-statement `SELECT` queries are allowed.
- Mutating statements such as `INSERT`, `UPDATE`, `DELETE`, `DROP`, `ALTER`, and `CREATE` are rejected.
- Results are capped at 50 rows.
- Tool output is returned to the model as structured JSON, then the model writes the final user-facing answer.

## Run With Docker Compose

Prerequisites:

- Docker Desktop.
- Internet access for pulling Docker images and Open-Meteo weather calls.

Start services:

```powershell
docker compose up --build -d
```

On first startup, Docker Compose automatically runs a one-shot
`ollama-pull` service that downloads `llama3.2:3b`. This can take a few
minutes. The model is stored in the `ollama-data` volume and reused on later
starts unless volumes are removed with `docker compose down -v`.

Check health:

```powershell
Invoke-WebRequest http://localhost:8080/health -UseBasicParsing
```

Swagger is available in development at:

```text
http://localhost:8080/swagger
```

Stop services:

```powershell
docker compose down
```

## Run Locally

Start Ollama locally and pull the model:

## Example Tasks

Because the final wording is produced by the local model, exact text can vary. The expected behavior and trace shape should remain consistent.

| Task | Expected answer | Expected trace |
|---|---|---|
| `Calculate 2 + 3 * 4` | Answer explains the result is `14`. | `TaskStarted`, `ModelDecision`, `ToolCall calculator`, `ToolResult calculator`, `FinalAnswer`, `TaskCompleted`. |
| `What is 15 percent of 68?` | Answer explains the result is `10.2`. | Uses `calculator` with an expression similar to `68 * 0.15`. |
| `Convert 10 kilometers to miles` | Answer is approximately `6.21 miles`. | Uses `unit_converter`. |
| `Convert 20 Celsius to Fahrenheit` | Answer is `68 Fahrenheit`. | Uses `unit_converter`. |
| `What is the current weather in London?` | Answer summarizes current temperature, condition, humidity/wind if available. | Uses `weather`, then model summarizes the returned weather payload. |
| `What is the cost of iPhone 17?` | Answer says the seeded catalog price is `$1,199.00`. | Uses `database_query` with a `SELECT` over `products`. |
| `Which products are low in stock?` | Answer lists products with small `stock_quantity`, such as `USB-C Cable` and `Laptop Stand`. | Uses `database_query`. |
| `What is the total value of order 1001?` | Answer calculates the order total from seeded `order_items`. | Uses `database_query` with a join or aggregate. |

## Observability Data

The SQLite database is configured by `ConnectionStrings:AgentDatabase` and defaults to:

```text
data/agent-observability.db
```

In Docker Compose it is stored in the `agent-data` volume:

```text
/app/data/agent-observability.db
```

The catalog bonus database is configured separately:

```text
CatalogDatabase:ConnectionString = Data Source=data/catalog.db
```

In Docker Compose:

```text
/app/data/catalog.db
```

Each task stores:

- Input task text.
- Model name.
- Status and final answer/error.
- Created, started, and completed timestamps.
- Total latency.
- Prompt, completion, and total token usage.
- Ordered trace events with tool names, arguments, results, latency, and errors where relevant.

## Notes And Assumptions

- The default LLM provider is local Ollama, so no external LLM API key is required.
- Weather uses Open-Meteo, which does not require an API key.
- Currency conversion is intentionally not implemented because it requires live exchange-rate data.
- The optional `database_query` bonus is implemented.
- Multi-turn context and frontend bonuses are not implemented.
